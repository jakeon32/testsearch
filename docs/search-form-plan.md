# 서치폼 기획서 — 화이트라벨 듀얼 모드

> 프로젝트: rideus 셔틀버스 예약 서치폼
> 작성일: 2026-03-16
> 버전: v2.0

---

## 1. 개요

### 1.1 목적

화이트라벨 서비스 특성에 맞게, 관리자가 **서치폼 모드를 선택**할 수 있도록 한다.

- **Mode A** — 노선 중심 (출발지/도착지 → 날짜) : 기존 방식
- **Mode B** — 날짜 중심 (날짜 → 출발지/도착지) : 신규

### 1.2 배경

| 서비스 유형 | 적합 모드 | 이유 |
|------------|----------|------|
| 다수 노선 (10+), 고정 스케줄 | **Mode A** (노선 중심) | 노선이 많아 먼저 좁혀야 탐색 가능 |
| 소수 노선 (1~5), 비정기 운행 | **Mode B** (날짜 중심) | "언제 운행하는지"가 1차 관심사 |
| 이벤트/시즌 셔틀 | **Mode B** (날짜 중심) | 특정 날짜 운행 여부가 핵심 |
| 통근/통학 셔틀 | **Mode A** (노선 중심) | 노선 고정, 날짜보다 경로가 중요 |

## 2. 관리자 설정

### 2.1 설정 항목

```
화이트라벨 관리자 패널
└── 예약 설정
    └── 서치폼 모드
        ├── ● 노선 중심 (Mode A) — 출발지/도착지를 먼저 선택
        └── ○ 날짜 중심 (Mode B) — 날짜를 먼저 선택
```

### 2.2 설정 데이터

```json
{
  "searchForm": {
    "mode": "route" | "date",
    "defaultTripType": "oneway" | "roundtrip",
    "maxPassengers": 10
  }
}
```

### 2.3 설정 반영 방식

- 관리자가 모드 변경 → 프론트엔드 서치폼 컴포넌트가 모드에 따라 분기 렌더링
- 동일한 데이터 소스 (`bookingDownDateByFrom`, `fromAvailableRoute` 등) 사용
- **검색 결과 페이지는 모드 무관하게 동일** (전달 파라미터 동일)

## 3. 공통 요소

### 3.1 공통 데이터 구조

| 키 | 설명 | 예시 |
|----|------|------|
| `fromAvailableRoute` | 출발지ID → 도착지ID[] | `{"302": [5], "100": [200, 201]}` |
| `bookingDownDateByFrom` | 출발지→도착지별 운행일자 | `{"302":{"5":["2026-03-16",...]}}` |
| `bookingUpDateByFrom` | 도착지→출발지별 귀환일자 | `{"5":{"302":["2026-03-16",...]}}` |
| `bookingDownDate` | 전체 운행 가능일 (모든 노선 합집합) | `["2026-03-16", ...]` |
| `locations` | 지점ID → 지점명 매핑 | `{"302": "서울 강남", "5": "에버랜드"}` |

### 3.2 공통 UI 요소

| 요소 | 설명 |
|------|------|
| 편도/왕복 토글 | 양 모드 공통. 기본값은 관리자 설정 |
| 인원 선택 | +/- 카운터, 기본 1명 |
| 검색 버튼 | 필수 항목 완료 시 활성화 |

### 3.3 공통 검색 파라미터

```
/search-results?
  type=oneway|roundtrip
  &departure_date=2026-03-16
  &return_date=2026-03-18        // 왕복 시에만
  &from_id=302
  &to_id=5
  &passengers=2
```

→ **검색 결과 페이지는 모드 무관하게 동일한 파라미터를 받는다.**

## 4. Mode A — 노선 중심 서치폼 (기존)

### 4.1 플로우

```
[편도/왕복] → [출발지 선택] → [도착지 선택] → [출발 날짜] → (귀환 날짜) → [인원] → [검색]
```

### 4.2 단계별 동작

| Step | 동작 | 필터링 |
|------|------|--------|
| 1 | 편도/왕복 선택 | — |
| 2 | 출발지 선택 | 전체 출발지 목록 |
| 3 | 도착지 선택 | `fromAvailableRoute[출발지ID]` 기준 |
| 4 | 출발 날짜 선택 | `bookingDownDateByFrom[출발지][도착지]` 기준, 운행일만 활성 |
| 5 | 귀환 날짜 선택 (왕복) | `bookingUpDateByFrom[도착지][출발지]` 기준, 출발일 이후만 |
| 6 | 인원 선택 | — |
| 7 | 검색 | 결과 페이지 이동 |

### 4.3 와이어프레임 (데스크톱)

```
┌────────────────────────────────────────────────────────────────────────┐
│  [편도 | 왕복]                                                          │
│                                                                        │
│  ┌────────┐  ┌────────┐  ┌──────────┐  ┌──────────┐  ┌──────┐        │
│  │ 출발지  │  │ 도착지  │  │ 출발 날짜  │  │돌아오는날짜│  │ 인원  │        │
│  │ 서울  ▼ │  │에버랜드▼│  │ 3/16(월) ▼│  │ 3/18(수) ▼│  │ ➖2➕ │        │
│  └────────┘  └────────┘  └──────────┘  └──────────┘  └──────┘        │
│                                                                        │
│  ┌──────────────┐                                                      │
│  │   검색하기     │                                                      │
│  └──────────────┘                                                      │
└────────────────────────────────────────────────────────────────────────┘
```

## 5. Mode B — 날짜 중심 서치폼 (신규)

### 5.1 핵심 콘셉트

**"날짜 먼저, 노선은 자동"**

- 날짜 선택 → 해당일 운행 노선의 출발지만 표시
- 출발지 1개면 자동 선택
- 도착지 1개면 자동 선택 (선택 UI 생략)
- 최적의 경우: 날짜 + 인원만 선택하면 검색 가능

### 5.2 플로우

```
[편도/왕복] → [출발 날짜] → (귀환 날짜) → [출발지] → [도착지] → [인원] → [검색]
                                          (자동선택)   (자동선택)
```

### 5.3 단계별 상세

#### Step 1: 편도/왕복 선택

| 항목 | 설명 |
|------|------|
| **기본값** | 관리자 설정 (`defaultTripType`) |
| **UI** | 토글 버튼 (편도 / 왕복) |
| **동작** | 왕복 선택 시 귀환 날짜 입력 영역 활성화 |

#### Step 2: 출발 날짜 선택

| 항목 | 설명 |
|------|------|
| **UI** | 캘린더 (모바일: 풀스크린 / 데스크톱: 드롭다운) |
| **활성 날짜** | `bookingDownDate` — 전체 노선 중 1개라도 운행하는 날짜만 |
| **비활성 날짜** | 운행 없는 날짜는 회색 + 선택 불가 |

#### Step 2b: 귀환 날짜 선택 (왕복 시)

| 항목 | 설명 |
|------|------|
| **표시 조건** | 왕복 선택 시에만 |
| **활성 날짜** | 출발일 이후, 귀환 스케줄이 있는 날짜 |
| **재검증** | 출발지/도착지 확정 후 → 해당 노선 귀환 가능일로 재필터링 |
| **당일 왕복** | 출발일 = 귀환일 선택 가능 |

#### Step 3: 출발지 선택

| 항목 | 설명 |
|------|------|
| **옵션 생성** | `bookingDownDateByFrom`에서 선택 날짜가 포함된 출발지ID 추출 |
| **1개인 경우** | 자동 선택 + 확정 표시 (드롭다운 생략) |
| **2개 이상** | 드롭다운에서 선택 |

```javascript
// 출발지 필터링 로직
availableDepartures = []
for (fromId in bookingDownDateByFrom) {
  for (toId in bookingDownDateByFrom[fromId]) {
    if (bookingDownDateByFrom[fromId][toId].includes(selectedDate)) {
      availableDepartures.push(fromId)
      break
    }
  }
}
```

#### Step 4: 도착지 선택

| 항목 | 설명 |
|------|------|
| **옵션 생성** | `fromAvailableRoute[출발지ID]` 중 선택 날짜에 운행하는 도착지만 |
| **1개인 경우** | **자동 선택** — 선택 UI 생략, "도착지: OO" 확정 표시 |
| **2개 이상** | 드롭다운에서 선택 유도 |

```javascript
// 도착지 필터링 로직
availableArrivals = fromAvailableRoute[selectedDepartureId]
  .filter(toId =>
    bookingDownDateByFrom[selectedDepartureId][toId].includes(selectedDate)
  )

if (availableArrivals.length === 1) {
  autoSelectArrival(availableArrivals[0])
}
```

#### Step 5: 인원 선택

| 항목 | 설명 |
|------|------|
| **기본값** | 1명 |
| **UI** | +/- 카운터 |
| **범위** | 1 ~ `maxPassengers` (관리자 설정) |

#### Step 6: 검색

| 항목 | 설명 |
|------|------|
| **활성 조건** | 편도: 출발날짜 + 출발지 + 도착지 + 인원 |
|               | 왕복: + 귀환날짜 |
| **동작** | 검색 결과 페이지로 이동 (공통 파라미터) |

## 6. 데이터 흐름 비교

### Mode A (노선 중심)

```
출발지 선택 ──→ 도착지 필터 ──→ 날짜 필터 ──→ (귀환일 필터) ──→ 검색
     │              │              │
     │         fromAvailable   bookingDown
     │         Route[fromId]   DateByFrom
     │                         [from][to]
     ▼
  전체 출발지
```

### Mode B (날짜 중심)

```
날짜 선택 ──→ 출발지 필터 ──→ 도착지 필터 ──→ (귀환일 재검증) ──→ 검색
     │              │              │
     │         bookingDown    fromAvailable
     │         DateByFrom     Route[fromId]
     │         에서 해당일      + 해당일 운행
     │         포함 출발지      도착지만
     ▼
bookingDownDate
(전체 운행일)
```

## 7. 엣지 케이스 & 상태 처리

### 7.1 공통

| 상태 | 처리 |
|------|------|
| 전체 스케줄 없음 | "현재 예약 가능한 스케줄이 없습니다" |
| 왕복 귀환 가능일 없음 | "선택하신 조건에 귀환 스케줄이 없습니다" |

### 7.2 Mode B 전용

| 상태 | 처리 |
|------|------|
| 선택 날짜에 운행 노선 0개 | 해당 날짜 비활성 (선택 불가) |
| 출발지 1개 | 자동 선택 + "출발지: OO" 확정 표시 |
| 출발지 2개+ | 드롭다운 선택 유도 |
| 도착지 1개 | **자동 선택** + 확정 표시 (드롭다운 생략) |
| 도착지 2개+ | 드롭다운 선택 유도 |
| 귀환일 → 출발지/도착지 확정 후 불일치 | 귀환일 재검증, 불가 시 귀환일 리셋 + 안내 |

### 7.3 모드 전환 시

| 상태 | 처리 |
|------|------|
| 관리자가 모드 변경 | 프론트엔드 즉시 반영 (서버 설정 읽기) |
| 사용자가 폼 작성 중 모드 변경 | 발생 안 함 (관리자만 변경 가능, 사용자 세션 중 불변) |

## 8. 컴포넌트 설계 방향

### 8.1 구조

```
SearchForm (최상위)
├── config.mode → "route" | "date"
│
├── TripTypeToggle (공통)
│
├── [mode === "route"]
│   └── RouteFirstFlow
│       ├── DepartureSelect
│       ├── ArrivalSelect
│       ├── DatePicker (출발)
│       └── DatePicker (귀환)
│
├── [mode === "date"]
│   └── DateFirstFlow
│       ├── DatePicker (출발)
│       ├── DatePicker (귀환)
│       ├── DepartureSelect (자동선택 로직 포함)
│       └── ArrivalSelect (자동선택 로직 포함)
│
├── PassengerCounter (공통)
└── SearchButton (공통)
```

### 8.2 공유 컴포넌트

| 컴포넌트 | 공유 여부 | 비고 |
|----------|----------|------|
| `TripTypeToggle` | 공유 | 양 모드 동일 |
| `DatePicker` | 공유 | 활성 날짜 배열만 다르게 주입 |
| `DepartureSelect` | 공유 | 옵션 목록 생성 로직만 다름 |
| `ArrivalSelect` | 공유 | Mode B에서 자동선택 로직 추가 |
| `PassengerCounter` | 공유 | 양 모드 동일 |
| `SearchButton` | 공유 | 활성 조건 검사만 다름 |

→ **UI 컴포넌트는 공유, 필터링/자동선택 로직만 모드별 분기**
